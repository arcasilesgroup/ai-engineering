# Sub-Spec 003: Verify Governance Direct Import

## Scope

Replace subprocess call to `uv run ai-eng validate --json` in `verify_governance()` with direct Python import of `validate_content_integrity()`. Map each failed validation category to a `Finding` with proper severity and location detail, instead of the current single opaque CRITICAL finding. Update verify tests.

## Exploration

- `verify_governance()` at service.py:116-132 calls `_run(["uv", "run", "ai-eng", "validate", "--json"])`, checks returncode, adds single CRITICAL finding
- `validate_content_integrity(target)` returns `IntegrityReport` with `.checks: list[IntegrityCheckResult]`
- Each `IntegrityCheckResult` has: `category` (StrEnum), `name`, `status` (OK/WARN/FAIL), `message`, `file_path`
- Validator is purely read-only — zero side effects
- No circular import risk (`verify/` does not import `validator/` and vice versa)
- `FakeSubprocess` fixture stays for quality/security tests

## Plan

### Step 1: Add imports to verify/service.py
- `from ai_engineering.validator.service import validate_content_integrity`
- `from ai_engineering.validator._shared import IntegrityStatus`

### Step 2: Replace verify_governance() body
- Remove subprocess call to `_run(["uv", "run", "ai-eng", "validate", "--json"])`
- Call `validate_content_integrity(project_root)` directly
- Map `IntegrityStatus.FAIL` -> `FindingSeverity.CRITICAL` per check
- Map `IntegrityStatus.WARN` -> `FindingSeverity.MINOR` per check
- Use `check.category.value` as Finding category (e.g. "mirror-sync")
- Use `check.file_path` as Finding file location

### Step 3: Update tests/unit/test_verify_service.py
- Replace `TestVerifyGovernance` class (3 new tests: pass, fail->critical, warn->minor)
- Mock `validate_content_integrity` via monkeypatch instead of FakeSubprocess
- Update `TestVerifyPlatform.test_aggregates_findings_from_all_modes` to add monkeypatch for governance

### Step 4: Verify
- `pytest tests/unit/test_verify_service.py -v`
- `ruff check src/ai_engineering/verify/`
