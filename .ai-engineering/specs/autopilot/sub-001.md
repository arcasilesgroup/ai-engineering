# Sub-Spec 001: Provider/IDE Field Separation

## Scope

Add `AiProvidersConfig` sub-model to `ProvidersConfig` with `enabled`/`primary` fields. Migrate all consumers from `providers.ides` to `providers.ai_providers.enabled` for AI provider operations. Fix `add_provider()`/`remove_provider()` to actually write to manifest. Add migration in loader for legacy format. Update dogfooding manifest.yml.

## Exploration

- `ProvidersConfig` has only `ides: list[str]` with default `["claude_code"]` — no ai_providers field
- `add_provider()` reads `providers.ides` for duplicate check but NEVER writes to manifest
- `remove_provider()` same bug — reads but never writes
- `_KNOWN_IDES` and `_VALID_AI_PROVIDERS` already defined and disjoint in operations.py
- `InstallContext` already has separate `providers` and `ides` fields at runtime
- `ide_config.py` (installer + doctor) reads `providers.ides` treating all entries as AI providers
- `test_provider_commands.py` uses `manifest.ai_providers.enabled`/`.primary` — describes target API
- `providers.ides` must stay for IDE tooling (stack_ide.py) — kept for backward compat
- `ManifestConfig` at top level already has space for `ai_providers` field
- Migration: loader splits `providers.ides` using `_VALID_AI_PROVIDERS` set when `ai_providers` absent
- `update_manifest_field` requires key to pre-exist — need ruamel helper for new section

## Plan

### Step 1: Add AiProvidersConfig to manifest.py
- New model: `AiProvidersConfig(enabled: list[str], primary: str)`
- Add `ai_providers: AiProvidersConfig` field to `ManifestConfig` (top-level, not nested under providers)
- Keep `providers.ides` for backward compat (IDEs only going forward)

### Step 2: Update template + live manifest.yml
- Add `ai_providers:` section with `enabled` and `primary` to template manifest
- Add same to live `.ai-engineering/manifest.yml`

### Step 3: Add migration logic in loader.py
- In `load_manifest_config()`: if raw YAML has no `ai_providers` key, derive from `providers.ides` using `_VALID_AI_PROVIDERS` filter
- First entry becomes `primary`
- Remaining non-AI-provider entries stay in `providers.ides`

### Step 4: Fix operations.py add_provider/remove_provider
- `add_provider()`: check `ai_providers.enabled`, write via ruamel after template copy
- `remove_provider()`: add "not enabled" guard, "last provider" guard, update primary on promotion, write via ruamel
- Add `_write_ai_providers()` helper

### Step 5: Update provider.py CLI
- Switch all `.providers.ides` references to `.ai_providers.enabled`/`.ai_providers.primary`

### Step 6: Update ide_config.py (installer + doctor)
- Switch `providers.ides` reads to `ai_providers.enabled` for AI provider operations

### Step 7: Update lib/signals.py
- Switch line ~675 from `config.providers.ides` to `config.ai_providers.enabled`

### Step 8: Update installer/service.py
- After template copy during install, write `ai_providers` based on providers parameter

### Step 9: Update test_provider_cli.py
- Update `_FakeManifest` to expose `.ai_providers` with `enabled`/`primary`

### Step 10: Update test_manifest.py + test_yaml_write.py
- Add `TestAiProviders` class, migration test via loader, write tests for new fields
- Update `SAMPLE_MANIFEST` fixtures

### Step 11: Verify
- `pytest tests/unit/ tests/integration/ -x`
- `ruff check src/`
