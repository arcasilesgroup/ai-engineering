## Architecture

pipes-and-filters

spec-119 extends the existing per-task execution kernel (`dispatch → build-verify-review → artifact collection → board sync`) with a new filter stage `ai-evaluator` between `ai-build` (Generator) and `ai-verify` (static-gate dispatcher). Each agent is a filter that consumes a typed contract from the upstream stage and emits a typed contract to the downstream stage; the Context Output Contract from `ai-build` becomes the input contract for `ai-evaluator`, whose structured pass@k verdict envelope becomes additional context that `ai-verify` and `ai-review` can read. The pattern's "When to use" criteria match: independent processing stages, well-defined input/output contracts at each boundary, the ability to insert or remove filters without rewiring the chain, and observability emitted at each stage boundary. Cross-cutting concerns (telemetry through `eval_run` events, manifest-driven thresholds, structured lint envelopes) layer on top of the pipe-and-filter chain as observers and configuration data, not as new pipeline stages. No bounded module is introduced; the changes are surgical extensions to existing files (`observability.py`, `manifest.yml`, `audit-event.schema.json`, `execution-kernel.md`) plus two new canonical artefacts (`ai-evaluator.md` agent, `ai-eval-gate` skill) and one new schema (`lint-violation.schema.json`).

## Sequencing Notes

- Phase 3 (lint-as-prompt rollout) depends only on Phase 1 (T-1.9 governance sign-off) and is independent of the agent work in Phase 2. The DAG sequences T-3.1 after T-1.9; teams that want to parallelise can run Phase 2 and Phase 3 concurrently after Phase 1, joining at Phase 4.
- Phase 4 depends on both Phase 2 (T-2.7 verifies the agent path) and Phase 3 (T-3.8 verifies structured envelopes), because the gate skill emits structured violations through the compliance-trace surface and consumes the agent's verdict envelope.
- Each phase that adds or modifies skill, agent, or kernel files (Phase 2, Phase 3, Phase 4) ends with a `sync-mirrors` task to keep canonical and IDE mirrors aligned. Phase 1 does not need sync-mirrors because it touches schemas and Python helpers that live outside the IDE-mirror surface.
- Audit-schema changes (T-1.4) and emitter implementation (T-1.5) ship together with the kind addition (T-1.3) so the schema validator never sees an unrecognised `eval_run` event.
- Manifest schema changes (T-1.6) ship before any phase that reads manifest thresholds (Phase 4 reads `manifest.evaluation`); this prevents downstream phases from depending on an unvalidated section.
- The lint-as-prompt rollout (Phase 3) intentionally lands a legacy-prose fallback (T-3.5) so the migration is not a hard cut; the fallback emits a `framework_error` of kind `legacy_violation_prose_seen` to make residual call sites visible in telemetry and easy to track.
- The `--skip-eval-gate` escape hatch (T-4.3) is wired through the audit chain so any bypass is observable in `framework-events.ndjson`; this preserves the regulated-industry tamper-evident properties of the audit chain.
