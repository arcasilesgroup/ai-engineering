# Spec 021 — Governance + CI/CD Evolution

## Problem

ai-engineering has strong local governance and quality gates, but there is a gap between local readiness and repository-level operational readiness:

1. `ai-eng install` currently bootstraps content, state files, and git hooks, but it does not leave every project with CI/CD and AI PR review fully operational by default.
2. Branch protection/build validation policies are not automatically applied, leaving repositories partially governed unless manually configured.
3. Tool readiness remediation is Python-focused; VCS CLIs (`gh`, `az`) and system tools (`gitleaks`, `semgrep`) are not installed through a complete OS-aware flow.
4. Stack-aware CI generation is not complete enough for real mixed-stack repos (for example: .NET backend + React frontend + Azure services) where Python checks must not run when Python is not part of installed stacks.
5. Agent/skill surface has strategic gaps for long-term governance evolution: no orchestrator brain, no strategic navigator, no docs simplification mode, and no dedicated data-model/data-security skills.
6. Agent surface has duplication pressure (many potential "role agents") that should be solved with a small set of perspective agents plus domain references, not agent sprawl.

## Solution

Implement a governance and delivery evolution centered on four outcomes:

1. **Install-to-operational by default**: expand `ai-eng install` so repositories end installation with stack-aware CI/CD + AI PR review + branch policy enforcement attempts.
2. **API-first fallback reliability**: when VCS CLI tools cannot be installed (compliance restrictions) or policies cannot be applied (permission gaps), automatically fall back to direct API mode and generate exact manual setup guidance.
3. **Agent/skill evolution with minimal cognitive surface**: adopt an orchestrator-driven architecture, add navigator and specialized governance/docs/devops agents, remove role explosion, and add data-focused skills.
4. **Governance content efficiency**: introduce docs simplification and adaptive standards loops to keep governance concise, current, and auditable.

## Scope

### In Scope

- Create and register new agents: `orchestrator`, `navigator`, `devops-engineer`, `docs-writer`, `governance-steward`, `pr-reviewer`.
- Remove `codebase-mapper` as a standalone agent and fold mapping phase into `architect`.
- Add new skills: `workflows/self-improve`, `dev/data-modeling`, `review/data-security`, `docs/simplify`, `govern/adaptive-standards`.
- Remove `skills/patterns/**` category and migrate content into on-demand `references/` in consuming skills.
- Merge `quality/audit-report` output contract into `quality/audit-code`.
- Expand installer runtime with phases for tool installation, VCS authentication, pipeline generation, policy application, and verification.
- Add stack-aware CI/CD generation for GitHub Actions and Azure DevOps using installed stacks from manifest.
- Add AI PR review pipeline generation with high/critical findings mapped to merge-blocking outcomes.
- Add branch policy/build validation apply-by-API behavior with manual guide fallback.
- Extend `doctor`/readiness checks to include CI/CD and branch-policy readiness.
- Update all wrappers/pointers (Copilot + Claude) and manifest counters/registrations.
- Keep multi-model routing out of scope for implementation (revert to single-platform model behavior).

### Out of Scope

- MCP/A2A/ACP runtime interop implementation.
- External hosted AI reviewer service deployment.
- New telemetry collection.
- Rewriting core Python architecture unrelated to install/vcs/pipeline/governance changes.

## Acceptance Criteria

1. `ai-eng install` runs with expanded phases and reports READY / READY WITH MANUAL STEPS / FAILED.
2. Install attempts OS-aware installation for required tools by stack and chosen VCS provider.
3. If VCS CLI install/auth is blocked by environment policy, install switches to API fallback mode without aborting governance setup.
4. CI/CD files are generated for GitHub or Azure DevOps according to selected provider, not both.
5. CI/CD jobs are stack-aware and only include checks for stacks in `install-manifest.json`.
6. AI PR review pipeline is generated by default and integrated with governance/security checks.
7. High/critical findings from PR review path are represented as merge-blocking outcomes through required status checks/build validations.
8. Branch protections/policies are attempted via API; if blocked, a deterministic manual setup guide is generated in project workspace.
9. `doctor` can report CI/CD and branch-policy readiness state.
10. New agent set is registered and wrapper parity is maintained across `.claude/commands/agent/` and `.github/agents/`.
11. `navigator` agent is added as read-only strategic analysis agent using existing artifacts only.
12. Data domain is covered as skills (`dev/data-modeling`, `review/data-security`) and references (`database-patterns.md`) without introducing a database-only agent.
13. Docs simplification capability exists via `docs-writer` simplify mode and `docs/simplify` skill.
14. Adaptive standards lifecycle exists via `govern/adaptive-standards` skill.
15. Integrity checks pass after migration updates and template mirror remains synchronized.

## Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| D021-001 | Remove multi-model routing from implementation scope | Current platform tooling does not provide robust cross-model runtime orchestration in this repo flow; single-platform execution is deterministic and reliable today |
| D021-002 | Keep agent count small and perspective-based | Avoid 20+ near-duplicate "role agents"; preserve maintainability and stronger governance consistency |
| D021-003 | Add `navigator` instead of delivery-lead/product-manager agents | Navigator fits existing artifacts and strategic gap without requiring external process systems (Jira/sprints/people management) |
| D021-004 | Implement data capability as skills, not agent | Data work requires procedures and references, not a separate judgment persona |
| D021-005 | AI PR review is mandatory in generated CI/CD with high/critical merge blocking | Enforces consistent security and quality posture by default |
| D021-006 | API-first fallback is required when CLI tooling/policies are blocked | Prevents incomplete installations in restricted enterprise environments |
| D021-007 | Remove `patterns/` skill category and migrate to `references/` | Pattern catalogs are reference knowledge, not executable procedures |
